# 왜 Context Capsule을 만들었나 — 실험 근거

## 배경

바이브코딩(AI 코딩 도구에 작업을 통째로 맡기는 방식)이 빠르게 퍼지고 있다.  
근데 실제로 써보면 불편한 게 있다.

> "레포 README 포폴용으로 수정해줘"  
> → AI한테 파일을 어떻게 줘야 하지?  
> → 그냥 다 붙여넣으면 토큰 얼마야?  
> → 관계없는 파일까지 다 읽히면 오히려 결과가 나빠지지 않나?

이걸 실제 케이스 하나로 측정해봤다.

주의: 이 문서는 정식 벤치마크가 아니라, Context Capsule이 왜 필요한지 설명하기 위한 정성/정량 혼합 스팟 체크다.

---

## 실험 설계

**대상 레포**: 공공조달 수요 분석 Python 프로젝트 (실제 개발 완료 레포)  
**작업 요청**: "README를 포트폴리오용으로 수정해줘"  
**비교 방식**: 같은 작업을 두 가지 방식으로 AI에게 넘김

| 방식 | 설명 |
|---|---|
| **A. Raw** | 레포 파일을 직접 AI에게 입력 |
| **B. Capsule** | Context Capsule 실행 후 출력물만 AI에게 입력 |

**모델 3종 × 방식 2종 = 6개 에이전트** 동시 실행 (Haiku / Sonnet / Opus 계열)

## 측정 결과

### 토큰 수

| 범위 | 문자 수 | 예상 토큰 |
|---|---|---|
| 레포 전체 텍스트 파일 (86개) | 566,302자 | **~188,767 토큰** |
| Context Capsule 출력 | 10,601자 | **~3,533 토큰** |
| **감소율** | | **~98%** |

레포를 통째로 AI에 넘기는 건 현실적으로 불가능하다 (컨텍스트 한계 초과).  
실제로 사용자가 하는 건 "README만 넘기기" — 그것도 약 3,000토큰.  
Context Capsule은 비슷한 토큰으로 **8개 파일의 핵심 내용을 선별해서** 넘긴다.

이 수치는 로컬 문자/토큰 추정 기준이다. 실제 Claude/Codex/OpenAI 과금 토큰은 provider usage 또는 별도 토큰 분석기로 검증해야 한다.

### 품질 비교 — 핵심 발견

레포의 `README.md`에는 ML 정확도가 **98.6%** 로 적혀 있고,  
`docs/qa_defense.md`에는 **98.08%** (홀드아웃 검증 기준)가 적혀 있다.

| 방식 | Haiku | Sonnet | Opus |
|---|---|---|---|
| **Raw** | ❌ 98.6% | ❌ 98.6% | ✅ 98.08%* |
| **Capsule** | ✅ 98.08% | ✅ 98.08% | ✅ 98.08% |

> *Opus Raw는 프롬프트에 직접 수치를 명시해줬기 때문에 정확함

**이 실험에서 Capsule을 쓴 에이전트는 모두 정확한 수치를 찾아냈다.**  
이유: Context Capsule이 `docs/qa_defense.md`를 관련 파일로 자동 선별했기 때문.  
Raw 에이전트는 README만 봐서 오래된 수치를 그대로 썼다.

### 결과물 품질 (5점 만점, 주관 평가)

| 에이전트 | 30초 파악 | 기술역량 강조 | 수치 정확성 | 스토리 | 포폴 구조 | **총점** |
|---|:---:|:---:|:---:|:---:|:---:|:---:|
| Haiku + Raw | 3 | 3 | 2 | 1 | 2 | **11** |
| Haiku + Capsule | 4 | 4 | 5 | 3 | 3 | **19** |
| Sonnet + Raw | 4 | 4 | 2 | 2 | 4 | **16** |
| Sonnet + Capsule | 4 | 4 | 5 | 3 | 4 | **20** |
| Opus + Raw | 5 | 5 | 5 | 3 | 5 | **23** |
| Opus + Capsule | 5 | 5 | 5 | 5 | 5 | **25** |

**이 실험에서는 Capsule을 쓴 에이전트가 모든 모델에서 Raw보다 높은 점수를 받았다.**

## 무엇을 확인했나

1. **토큰 낭비 가능성**: 레포를 통째로 넘기면 현실적으로 불가능하거나 큰 비용이 든다.
2. **컨텍스트 품질 문제**: 오래된 README만 보면 더 정확한 QA 문서 수치를 놓칠 수 있다.
3. **파일 선별 비용**: 필요한 파일을 사람이 매번 고르는 건 번거롭고 빠뜨리기 쉽다.

Context Capsule은 이 세 가지를 자동으로 해결한다:
- 레포를 스캔해서 태스크와 관련 있는 파일만 추출
- 이 케이스에서는 토큰을 ~98% 줄이면서 더 정확한 컨텍스트 제공
- 사람이 검토하고 승인한 뒤 AI에 넘기는 Human-in-the-loop 구조 유지

---

## 누구한테 필요한가

- **AI 결제가 부담스러운 신입 개발자 / 1인 개발자**: 토큰을 아껴쓸 수 있음
- **중소기업 개발팀**: 팀원에게 작업을 넘길 때 컨텍스트 손실 방지
- **바이브코딩을 처음 도입하는 팀**: AI에게 뭘 얼마나 줘야 하는지 기준이 생김
