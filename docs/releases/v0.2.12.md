# Context Capsule v0.2.12

v0.2.12 is an Installer UX & First Run Hardening release.

The scope is intentionally narrow: this is still a terminal-backed Windows ZIP flow. The goal is not a no-terminal desktop app yet. The goal is that, when first run fails, a tester sees Korean guidance and has a log file to send back.

## What Changed

- `run_context_capsule.bat` now prints Korean failure guidance.
- `scripts/install_windows.ps1` writes `outputs/logs/install_*.log`.
- `scripts/run_dashboard.ps1` writes `outputs/logs/dashboard_*.log`.
- `scripts/context_capsule_cli.ps1` writes `outputs/logs/cli_*.log`.
- Python/venv/pip/dashboard startup failures now show next actions in Korean.
- Docs now tell testers to attach the newest log from `outputs/logs`.

## What This Is Not

- Not a PyInstaller `.exe`.
- Not a no-terminal desktop shell.
- Not a new Token Analyzer integration.

Those belong to later product packaging or token-measurement work.

## Why It Matters

KDT testers and first-time users may not know whether a failure came from Python, dependency install, virtualenv setup, port conflict, or the app itself.

v0.2.12 makes that failure path more reviewable:

```text
double-click run_context_capsule.bat
-> failure happens
-> Korean next actions are printed
-> outputs/logs contains the install/dashboard/CLI log
-> tester can send the log with feedback
```
