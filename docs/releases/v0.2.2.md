# Context Capsule v0.2.2

v0.2.2 adds the Beta Feedback Loop.

The goal is simple: after KDT testers try Context Capsule on their own projects, their feedback should not disappear into Discord chat. It should become saved evidence, reviewable patterns, next patch priorities, and regression test candidates.

## Release asset

```text
context-capsule-v0.2.2.zip
```

## What's new

- Added a Korean-first feedback form under the FastAPI result area.
- Added `POST /api/feedback` to save one feedback packet.
- Added `POST /api/feedback-review` to summarize saved feedback.
- Added CLI command `feedback-save`.
- Added CLI command `feedback-review`.
- Saved feedback now writes:
  - `FEEDBACK.md`
  - `feedback.json`
- Feedback review can produce:
  - common complaints
  - missed file cases
  - UI confusion points
  - token/risk explanation questions
  - next patch priorities
  - regression test candidates

## Safety

- Feedback text is treated as untrusted input.
- Secret-looking values are redacted as `[REDACTED_SECRET]`.
- Prompt-injection-like lines are redacted as `[REDACTED_PROMPT_INJECTION_LINE]`.
- Feedback Review is not teammate scoring.
- Feedback Review does not assign work, create branches, or create GitHub Issues automatically.

## Dashboard flow

```text
Generate a packet
-> check Overview / Files / Risk
-> fill "이 결과가 이상했나요?"
-> save feedback
-> open "피드백 리뷰"
-> generate next patch priorities
```

## CLI flow

```powershell
.\context_capsule_cli.bat feedback-save `
  --mode work `
  --project-name "shop-app" `
  --request "로그인이 모바일에서만 안돼" `
  --expected-file backend/auth/login.py `
  --actual-file README.md `
  --confusing-part "결과 탭에서 어디를 봐야 하는지 헷갈렸어요." `
  --output-dir outputs\feedback `
  --json

.\context_capsule_cli.bat feedback-review `
  --feedback-root outputs\feedback `
  --save `
  --json
```

## Why this matters

v0.2.1 made the local app easier to use. v0.2.2 makes beta testing easier to learn from.

Instead of manually rereading Discord comments, Context Capsule can now collect structured feedback and convert it into the next implementation queue.

## Known limits

- Feedback Review is rule-based.
- It does not decide whether a tester is right or wrong.
- It does not measure actual Claude/Codex/OpenAI provider usage.
- It should be used as a patch planning aid, not as an automated product manager.
