# Context Capsule — 실험 결과 한 장 요약

> "AI 코딩 비용을 줄이면서 정확도는 올린다"

---

## 문제

바이브코딩 워크플로에서 AI에게 레포를 넘기는 방식은 두 가지다.

| 방식 | 입력 토큰 | 결과 |
|---|---|---|
| **Raw** (전체 레포 덤프) | ~4K ~ 200K+ | 모델이 노이즈에 묻혀 오답 또는 추상적 답변 |
| **Context Capsule** | ~2,500 고정 | 관련 파일만 추출 → 정확한 파일·함수·수치 지목 |

---

## 실험 설계

- **모델**: Claude Haiku 4.5 / Sonnet 4.6 / Opus 4.8
- **레포**: dummy-repo (16파일) · procurement-logistics-ai (86파일) · rainbow-bridge (451파일)
- **태스크 유형**: 버그 수정 요청 · 수치 조회 · 마이그레이션 · 배포 설정 (총 9개 태스크)
- **채점**: 정답 파일명·함수명·수치 키워드 포함 여부 (한국어 동의어 포함)

---

## 핵심 결과

### 전체 정답률

| | Raw | **Context Capsule** |
|---|---|---|
| 정답률 | 20/39 (51.3%) | **76/90 (84.4%)** |
| 평균 입력 토큰 | ~38,000 | **~2,500** |
| 평균 토큰 절감 | — | **71.8%** |

### 모델별 CC 적용 후 성적

| 모델 | CC 정답률 | 비용 (2,500 tok 기준) |
|---|---|---|
| Haiku 4.5 | 26/27 (96%) | ~$0.002 |
| **Sonnet 4.6** | **25/27 (93%)** | **~$0.008** |
| Opus 4.8 | 25/27 (93%) | ~$0.038 |

**핵심: CC를 쓰면 모델 간 성능 차이가 거의 사라진다.**

Haiku가 1문제 높게 보이지만, 27개 샘플에서 1문제 차이는 통계적으로 큰 의미가 없다. 또한 이 실험은 파일명·함수명·수치 키워드 포함 여부로 채점했기 때문에, 짧고 직접적으로 답하는 Haiku가 약간 유리할 수 있다.

따라서 강조할 메시지는 "Haiku가 제일 좋다"가 아니라 **컨텍스트가 정리되면 Sonnet도 Opus와 거의 같은 수준으로 일한다**는 점이다.

### 대형 레포에서 Raw의 한계

procurement-logistics-ai (86파일, 107K 토큰) Raw 결과:

| 모델 | Raw 정답률 | 오답 포함 |
|---|---|---|
| Haiku | **0/9** | README 수치 98.6% 오답 반환 |

CC 적용 시 전 모델 8/9, 오답 없음.

---

## 결론

### "설계는 한 번, 기능 추가·버그 수정은 매일"

실제 개발 작업의 90%는 특정 파일 몇 개만 알면 되는 태스크다.
CC는 그 파일만 골라 넘기기 때문에 **모델이 더 비싸서가 아니라 컨텍스트가 정리돼서** 정확도가 올라간다. 실험에서는 Sonnet + CC와 Opus + CC가 거의 같은 수준으로 수렴했다.

### 검증된 수치 (2026-06-29)

```
CC 정답률:     84.4%   (Raw 대비 +33.1%p)
토큰 절감:     71.8%
실험 총비용:   $1.83   (Opus 포함, CC 덕에 Raw Opus 호출 불필요)
```
